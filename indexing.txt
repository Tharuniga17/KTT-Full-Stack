Indexing is used in SQL , to speed up the data retrieval, it will create a separate data structure that will store column values with pointers

instead of scanning the entire table it will jump directly to the required data 

without indexing we need to scan entire table 

most databases uses b-tree where the data is sorted , and search is logarithmic and quickly navigate between left and right

Types of indexing:

Primary index -> automatically created on primary key 

id int primary key 

Unique index:

it is also a unique key where no duplicates values are allowed

CREATE UNIQUE INDEX idx_email ON users(email);

composite index: -> it contains multiple unique keys 

Clustered data:

sorts actual table data  --> only one per table 

Non- clustered data: 

separate structure storing pointers --> can have multiple

disadvantage is it takes extra memory, without index it stores only one storage -> just a table data 

with index -> it created an extra structure like a b-tree so it contains both table data and index data

QUERY OPTIMIZATION:

database has a component called query optimizer  which decides whether we can use seq scan or index scan 


index will help optimizer to avoid full table scan , reduce data access and improve search speed 

with index optimization is O(log n)

Optimizer will decide the table size 
for small table -> seq scan 
for large table -> index scan 

analyse ---> will help the optimizer to choose the correct plan

Transactions:

it is a group of queries which ensures that the data is correct and consistent even if the system crashes

Begin -> it will start a transaction block, the changes are not permanent here 

Commit -> all the changes are permanent --> after commit we cannot undo 

Rollback --> it will cancel everything after begin --> go back to the original state

Savepoint --> it will create a temporary checkpoint, instead of full roleback it will visit the last checkpoint and undo the operations 

Transaction isolation levels data:

read uncommited : --> can see uncommited data , it is a dirty read 

read commited --> can see only commited data , it is safe and widey used 

serializable -> transactions behave like one by one execution 

ACID Properties -> it ensures that our data in database are safe , reliable and consistent 

A -> Atomicity (all or nothing) 

either transaction will happens fully or doesn't happen at all 

if second query fails the money wont lost, here payment either fully succeed or money stays in our account not half done

Consistency --> data remains in a valid state (some rules/ constraints will never break)

marks <= 100 

update students set marks = 150 where id = 1;

here marks should be lesser than 150 so db will rejects it

with consistency data remains correct and rules must be followed

Isolation(no interference)  --> multiple transactions should not disturb each other 

no two users can access at the same time 

with isolation each transaction will work independently 

durability --> once commited , data is permanently stored, even if system crashes 

commit --> data will be stored in disk/log 

without durability --> power goes off, data lost 

with durability --> data stays safe, even after crash 

select transaction isolation level repeatable read;

select transaction isolation level serializable;

WINDOW CALCULATIONS:

it is used when you want calculation without grouping the rows

it will help by not losting the data , it contains partition by --> grouping(like group by but it wont collapse)

order by --> ordering inside each group

Row_number() --> assigns a unique sequence number to each row 

it will assign numbers quickly, 

select name, marks ROW_NUMBER() OVER (order by marks desc) as runfrom student;

Rank() -> it will give same rank for ties

but skips the next number

select name, marks RANK() OVER (order by marks desc) as rn from student;

here if two students perform same rank, it would assign same number, like 2 2 but it would skip 3 and moves to 4

Output:
name	marks	rnk
Arun	95	1
Bala	90	2
Chitra	90	2
Devi	85	4

👉 Notice: 3 is skipped ❌

DENSE_RANK() -> it is same as rank but does not skip numbers 

select name,marks DENSE_RANK() OVER (order by marks DESC) AS drnk from student;

name	marks	drnk
Arun	95	1
Bala	90	2
Chitra	90	2
Devi	85	3

LEAD() --> it will gets you next row value

select name,marks, LEAD(marks) OVER (order by marks desc) as next_marks from student;

name	marks	next_marks
Arun	95	90
Bala	90	90
Chitra	90	85
Devi	85	NULL

👉 Shows next row value

Lag() --> gets you previous row value 

SELECT name, marks,
       LAG(marks) OVER (ORDER BY marks DESC) AS prev_marks
FROM student;

name	marks	prev_marks
Arun	95	NULL
Bala	90	95
Chitra	90	90
Devi	85	90

👉 Shows previous row value
